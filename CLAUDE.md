Heading on Test is the full 12-bit range — degrees = `(heading * 360) >> 12`. `playerSelfPosStruct` u32 at offset 10 packs `signed deltaHeading:10` + **1-bit gap** + `unsigned heading:12` + `unsigned padding:9` (NOT legacy's 10+12+10); `animation` byte is at offset 18, "dri" marker at 19-21. Test wire layouts often add a 1-bit gap relative to legacy live — when a bit-stream decoder produces values pinned at int16 boundary, doubled-mod-360, or similar 1-bit-off symptoms, try shifting one field by 1 bit first. Wire-field re-audits need **3+ anchor points** before committing. Single /loc check can match a bad scale/sign that happens to agree at one position; three positions at clearly different coordinates expose constant-cluster bugs and sign flips. Workflow: `--record-vpk` while walking to 3 named NPCs in distinct parts of the zone, `/loc` next to each, then scan every bit-shift × width × scale combination against all three targets (Python brute-force, see 2026-05-14 spawnStruct session). EQ's `/loc` command displays `(-grid_y, -grid_x, grid_z)` — axes swapped AND negated vs the grid coords ShowEQ shows on the map. When mapping wire bytes anchored against `/loc`, the wire field labeled "x" in /loc semantics belongs at the struct's `y` slot (and vice-versa) so ShowEQ's existing `fillPos` negation gives the grid back. Symptom of getting this wrong: ShowEQ x/y values are swapped AND/OR sign-flipped from what the in-game grid map shows.

`BitStream::readInt(n)` (`netstream.cpp`) is sign-and-magnitude (sign bit first, then n-1 magnitude bits), NOT two's complement. **When diagnosing this parser via Python or any external decoder, MIRROR sign-and-magnitude exactly** — using a 2C decoder will make correct field widths look broken (values pin near int16 boundary) and a wrong wider field width look correct (values look reasonable but are ~2× the true value). A "decoded value is exactly -2× the correct value" symptom is the canonical SM-vs-2C / off-by-one-bit signature.
Test posData wire layout (shared by `spawnStruct.posData[6]` and `playerSpawnPosStruct`'s 5-word position block, audited 2026-05-05): **z at bits 0-18 of word 2** (×8 wire scale), **y at bits 13-31 of word 3**, **x at bits 13-31 of word 4**; words 0/1/5 hold state-flags / unused. Heading + delta* are NOT in this block on Test (they ride OP_MobUpdate / OP_NpcMoveUpdate). Diagnostic-symptom map: spawns rendering as a horizontal line at one Y → spawnStruct posData wrong (static spawns); spawns "teleporting to NW then snapping back" → some moving-position opcode wrong while another is right; "exactly -2× correct" pattern → SM-vs-2C / off-by-one-bit shift.

`EQPacketFragmentSequence::addFragment` (`packetfragment.cpp`) is INTENTIONALLY permissive: trust the first-fragment's declared total length, allocate it, let Linux overcommit absorb 4GB-sized junk allocations silently. Adding a sanity cap (e.g. "size > maxPacketSize → drop") cascades — a dropped first-fragment leaves the next legitimate continuation to be misread as a NEW first-fragment with garbage size, and so on, silently losing real downstream data. The original `seqFatal` on continuation overflow is the right default; if multibox / stream-clobber crashes need handling, fix at the session/stream layer (key validation, pinning), not here. Bit hard on 2026-05-16 (commits `8687904` / `400310d`, both reverted).
