`SessionAdapter::sendSnapshot` sorts spawns by `(id, name)` — EQ reuses ids across spawn types (a Door and a NPC can both have id 199), so id-only sort leaves duplicate-id pairs in QHash-randomized order and tier-2 goldens flap.

Heading on Test is the full 12-bit range — degrees = `(heading * 360) >> 12`. `playerSelfPosStruct` u32 at offset 10 packs `signed deltaHeading:10` + **1-bit gap** + `unsigned heading:12` + `unsigned padding:9` (NOT legacy's 10+12+10); `animation` byte is at offset 18, "dri" marker at 19-21. Test wire layouts often add a 1-bit gap relative to legacy live — when a bit-stream decoder produces values pinned at int16 boundary, doubled-mod-360, or similar 1-bit-off symptoms, try shifting one field by 1 bit first. Caveat: this exact "off-by-one-bit" symptom can ALSO be a sign-and-magnitude vs two's-complement reader mismatch (see BitStream::readInt note); rule out that first.
`BitStream::readInt(n)` (`netstream.cpp`) is sign-and-magnitude (sign bit first, then n-1 magnitude bits), NOT two's complement. **When diagnosing this parser via Python or any external decoder, MIRROR sign-and-magnitude exactly** — using a 2C decoder will make correct field widths look broken (values pin near int16 boundary) and a wrong wider field width look correct (values look reasonable but are ~2× the true value). A "decoded value is exactly -2× the correct value" symptom is the canonical SM-vs-2C / off-by-one-bit signature.
Test posData wire layout (shared by `spawnStruct.posData[6]` and `playerSpawnPosStruct`'s 5-word position block, audited 2026-05-05): **z at bits 0-18 of word 2** (×8 wire scale), **y at bits 13-31 of word 3**, **x at bits 13-31 of word 4**; words 0/1/5 hold state-flags / unused. Heading + delta* are NOT in this block on Test (they ride OP_MobUpdate / OP_NpcMoveUpdate). Diagnostic-symptom map: spawns rendering as a horizontal line at one Y → spawnStruct posData wrong (static spawns); spawns "teleporting to NW then snapping back" → some moving-position opcode wrong while another is right; "exactly -2× correct" pattern → SM-vs-2C / off-by-one-bit shift.

Test chat-text dispatch: OP_SimpleMessage packs `(targetSpawnId@0, color@4, spellId@8)`; OP_FormattedMessage packs `(target@0, formatId@4, color@8, argCount@9, padding@10, args[]@11)` where each arg is a 16-byte preamble + NUL-terminated UTF-8 string. Disambiguator: `target == 0x09e2` → system broadcast (eqstr/dbstr template path); `target` = spawn ID → spell-effect on that spawn (route to `spells_us_str.txt` column by self-vs-other check vs `m_player->id()`). The legacy "messageFormat is the eqstr key" reading is wrong on Test for spawn-targeted fires; format collides hard with spell IDs (eqstr 5265="yards" vs spell 5265="Pious Light"; eqstr 6000="Your guild has received %1 favor..." vs spell 6000="You are covered in dancing flames").
EQ ships 3 text tables; all 3 live under `/usr/local/share/showeq/` for the data-location cascade: `eqstr_us.txt` (template-substitution, %N args), `spells_us_str.txt` (caret-delim per-spell columns: CASTERMETXT/CASTEROTHERTXT/CASTEDMETXT/CASTEDOTHERTXT/SPELLGONE), `dbstr_us.txt` (two-keyed `id1^id2^text`; only `id2=0` is safe for blind chat fallback — other id2s are tooltip/spell-description content needing client-side substitution).
